# 安全分析标准参考文档

> 本文档定义 Web UI 安全侦察中阶段 6 的分析标准。
> 可根据需要扩展分析维度。

---

## 维度 1：高危功能识别

识别前端中涉及高风险业务功能的操作。

### 检查清单

| # | 检查项 | 风险等级 | 说明 |
|---|--------|---------|------|
| 1.1 | 文件上传功能 | 🟠 高危 | 是否存在 `<input type="file">` 或文件选择组件；上传的 API 端点；文件类型/大小是否有限制（以及限制在前端还是后端） |
| 1.2 | 文件下载功能 | 🟡 中危 | 下载 API 端点；是否存在路径遍历风险（如文件名直接拼接路径）；响应处理方式 |
| 1.3 | 数据删除功能 | 🔴 严重 | DELETE 请求；批量删除；是否有二次确认；软删除还是硬删除 |
| 1.4 | 权限变更功能 | 🔴 严重 | 修改用户角色、权限的操作；是否存在越权风险 |
| 1.5 | 资金/交易操作 | 🔴 严重 | 涉及金额的操作（支付、转账、提现）；金额参数是否可篡改 |
| 1.6 | 用户状态变更 | 🟠 高危 | 启用/禁用用户；锁定/解锁账户 |
| 1.7 | 配置修改功能 | 🔴 严重 | 系统配置修改；可能影响全局的设置 |
| 1.8 | 二次认证（2FA） | — | 登录/高危操作是否存在二次认证环节。如存在，记录触发条件（登录、支付、删除等），是 TOTP/SMS/生物识别/其他；如不存在，无需特别提及 |
| 1.9 | 二次编程/动态执行 | 🔴 严重 | 应用是否允许动态代码执行，如：自定义公式/脚本、规则引擎、模板注入、宏/脚本上传、SQL 查询面板、代码在线编辑等。这类功能即使有权限控制也属于高危 |
| 1.10 | 上传签名机制 | 🟠 关注 | 文件上传是否要求签名/Token（如 STS 临时凭据、Presigned URL、上传凭证）。缺少签名机制意味着任意用户可上传任意文件 |
| 1.11 | 上传文件格式 | 🟠 关注 | 上传功能允许的文件类型。重点关注是否允许可执行文件/软件包：`.exe`, `.msi`, `.sh`, `.bat`, `.ps1`, `.apk`, `.dmg`, `.pkg`, `.AppImage`, `.bin`, `.run`, `.deb`, `.rpm`, `.jar`, `.war` 等。也关注是否允许 `zip/rar/7z` 等压缩包（内部可能包含可执行文件） |

## 维度 2：输入参数校验分析

分析前端对用户输入的校验逻辑，识别校验缺失或可绕过。

### 检查清单

| # | 检查项 | 说明 |
|---|--------|------|
| 2.1 | 前端是否对输入做类型校验 | 如 `type="number"`、`type="email"` 等 HTML5 校验 |
| 2.2 | 前端是否对输入做长度/格式校验 | `minLength`、`maxLength`、`pattern` 属性或 JS 校验逻辑 |
| 2.3 | 校验仅在前端进行 | 若仅前端校验而后端不校验，则存在绕过风险 |
| 2.4 | 输入是否直接拼接到 URL/请求中 | 可能存在注入风险（SQL 注入、NoSQL 注入等） |
| 2.5 | 是否有防 XSS 措施 | React 默认转义 OK，但 `dangerouslySetInnerHTML`、`v-html` 等需关注 |

## 维度 3：凭据/PII 输入识别

识别应用中涉及敏感信息输入的位置。

### 检查清单

| # | 检查项 | 说明 |
|---|--------|------|
| 3.1 | 密码输入 | `<input type="password">`；是否有明文显示选项 |
| 3.2 | 认证凭据 | API Key、Token、Access Key/Secret Key 等输入 |
| 3.3 | 个人身份信息 | 身份证号、手机号、邮箱地址、家庭住址等 PII 输入 |
| 3.4 | 支付信息 | 银行卡号、CVV、支付密码等 |
| 3.5 | 敏感信息传输 | 凭据是否通过 HTTPS 传输；请求体中是否包含明文敏感信息 |

## 维度 4：URL/地址配置 — SSRF 风险

识别前端中可能引发 SSRF（服务端请求伪造）攻击的配置点。

### 检查清单

| # | 检查项 | 说明 |
|---|--------|------|
| 4.1 | 用户可配置 URL | 系统配置中的站点 URL、回调 URL、Webhook URL 等 |
| 4.2 | 外部服务 Endpoint | 存储服务 Endpoint、外部 API 地址等 |
| 4.3 | 代理/转发配置 | 请求代理地址、API 网关地址等 |
| 4.4 | 图片/资源 URL | 用户头像 URL、封面图 URL 等（后端是否 fetch） |
| 4.5 | 导入/导出 URL | 数据导入导出的外部地址配置 |

### SSRF 风险判断标准

- URL 以用户输入或配置为来源，且由后端发起请求 → 🔴 严重
- URL 可指向内网地址 → 🔴 SSRF 可被利用
- URL 来自用户输入但仅前端使用 → 🟡 低风险（但仍需关注 XSS）

## 维度 5：协议配置安全

分析应用中涉及协议配置的部分及其默认安全性。

### 检查清单

| # | 检查项 | 说明 |
|---|--------|------|
| 5.1 | 协议是否可配置 | 如 HTTP/HTTPS、TLS 版本、加密算法等 |
| 5.2 | 默认协议是否安全 | 默认使用 HTTPS？默认使用 TLS 1.2+？ |
| 5.3 | 是否存在协议降级风险 | 是否允许从安全协议降级到不安全协议（如 HTTPS→HTTP） |
| 5.4 | 是否允许不安全的协议选项 | 如 SMTP 允许无加密、允许 HTTP 回退 |
| 5.5 | WebSocket 协议 | WSS 是否强制？是否允许未加密的 WS？ |

## 维度 6：错误处理与信息泄露

分析前端错误处理机制，识别输入约束条件和潜在的信息泄露风险。

### 检查清单

| # | 检查项 | 说明 |
|---|--------|------|
| 6.1 | 前端校验规则提取 | 从 `rules`、`validators`、`yup/zod schema`、`pattern` 属性等中提取所有输入约束条件。这些约束揭示了后端预期的输入格式，对构造安全测试 payload 有直接参考价值 |
| 6.2 | 错误信息颗粒度 | 区分通用错误（"操作失败"）和具体错误（"用户名已被注册"）。具体错误信息可能导致用户枚举或信息泄露 |
| 6.3 | API 错误透传 | 后端错误信息是否直接展示给前端用户。如 `catch(e => showError(e.response.data.message))` 可能泄露 SQL 错误、堆栈信息、文件路径等 |
| 6.4 | 认证错误区分 | 登录/认证时的错误信息是否区分"用户不存在"和"密码错误"。区分这类错误可被用于用户名枚举攻击 |
| 6.5 | 错误反馈渠道 | 错误通过什么方式反馈（弹窗、行内校验、toast、控制台、静默）。静默失败可能隐藏安全事件 |
| 6.6 | 敏感信息在错误中泄露 | 错误信息/调试信息中是否包含：堆栈追踪、SQL 查询、文件路径、Token/凭据片段、API 密钥、内网地址等 |

### 分析示例

```yaml
登录页面错误处理分析:
  - 用户名空: "请输入用户名" (规则: {required: true}) → 前端校验
  - 密码空: "请输入密码" (规则: {required: true}) → 前端校验
  - 登录失败: "用户名或密码错误" → 通用错误 ✅ (不区分用户是否存在)
  - 捕获方式: catch → message.error → 通用提示

用户管理错误处理分析:
  - 创建用户邮箱格式: type:'email' → 前端校验
  - 密码长度: min:6 → 前端校验 (约束: 至少 6 位)
  - API 错误: "操作失败" → 通用错误 ✅
  - 但存在 catch(e => message.error(e.response.data.message)) → ⚠️ 后端错误可能透传

文件上传错误处理分析:
  - accept: "image/*,.pdf,.doc,.docx" → 前端文件类型限制
  - 上传失败: "上传失败" → 通用错误 ✅
  - 无文件大小校验 → 前端无限制
```
